## 第 6 次进展

### 对应任务

`T06 抽取模型 client 与异步批处理`

### 工作内容

1. 将真实 provider 的调用实现从手写 `httpx` 非流式请求切换为 `openai` SDK 的 `chat.completions.create(..., stream=True)` 流式聚合。
2. 统一将 `instruction + input_text` 合并为单条 `user` 消息，不再依赖系统提示词。
3. 为 SiliconFlow 的 `DeepSeek-R1` 增加兼容参数：
   - `temperature=0.6`
   - `top_p=0.95`
   - `extra_body.thinking_budget=128`
   - 在 `JSON / 三元组 / Cypher` 等结构化指令场景下增加 `max_tokens=256`
4. 将 `source_records.error_message` 从单一错误码扩展为带诊断信息的字符串，包含：
   - 错误码
   - 异常类型
   - 请求 URL
   - 模型名
   - 状态码（如有）
   - 响应体前缀（如有）
5. 在 `backend/requirements.txt` 中新增 `openai==2.28.0` 依赖。
6. 对 SiliconFlow 做了独立低层验证，分别测试了：
   - `DeepSeek-R1`
   - `DeepSeek-V3.2`
   - `Qwen/Qwen2.5-72B-Instruct`

### 验收情况

- `PYTHONUNBUFFERED=1 .venv/bin/python -m compileall app` 执行通过。
- 新版 [`llm_client.py`](/Users/aoao/Desktop/毕业论文/领域知识图谱构建和问答系统2.0/backend/app/services/llm_client.py) 直连 SiliconFlow 成功。
- `DeepSeek-R1` 在新版 client 下不再出现原先的 `RemoteProtocolError('Server disconnected without sending a response.')` 级别失败，能够返回成功结果。
- `DeepSeek-R1` 虽能返回，但输出仍偏解释型，不稳定满足“只输出 JSON 数组”。
- `DeepSeek-V3.2` 在相同抽取指令下约 `9.74s` 返回，`output` 直接是合法 JSON 数组，且没有 `reasoning_content` 混入。
- 结论：当前项目的抽取场景推荐优先使用 `deepseek-ai/DeepSeek-V3.2`，不建议继续用 `deepseek-ai/DeepSeek-R1` 做结构化三元组抽取。
