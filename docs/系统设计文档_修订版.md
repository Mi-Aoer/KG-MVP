# 通用领域知识图谱构建与问答系统

## 系统设计文档（MVP / 面向编码）

版本：V1.1
更新日期：2026-03-18
目标：以最少功能、最少耦合、最少代码量，实现以下主链路：

**上传 txt → 按行切分 → 调用抽取模型 → 展示/修正结果 → 生成并维护 schema → 导入 Neo4j → 基于图谱问答**

------

## 1. 设计目标与约束

### 1.1 目标

一期系统只解决 4 件事：

1. 让用户上传 txt 批量抽取三元组。
2. 让用户修正抽取结果，并从结果中生成通用 schema。
3. 让用户把三元组导入同一个 Neo4j 数据库中的项目级图谱。
4. 让用户在当前项目图谱上发起单轮问答。

### 1.2 技术约束

- 后端：**FastAPI**
- 前端：**Vue 或 React 单页应用**
- 元数据存储：**SQLite**
- 图数据库：**Neo4j**
- 部署方式：**本地单机部署**
- 抽取执行：**异步批处理 + 前端轮询**
- 不做登录鉴权
- 不做复杂图谱可视化
- 不做自动同步删除
- 不做多轮对话
- 不做消息队列，不做分布式事务

### 1.3 实现原则

- 单体应用，不拆微服务
- 一个批次同一时间只允许一个抽取任务
- 后端默认单进程单 worker 运行
- 所有外部模型调用都经由后端代理
- 所有问答必须严格基于图谱结果，不允许自由发挥
- LLM Client 采用“统一接口 + provider / 模型适配器”设计，不把所有 OpenAI-compatible provider 视为完全等价接口
- 抽取场景默认将 `instruction + input_text` 合并为单条 `user` 消息，`system` 消息仅作为可选 provider 策略
- 真实 provider 允许使用官方 / 兼容 SDK，以及 `stream=True` 后聚合最终输出

## 2.2 架构分层

这个设计保留了论文里“存储层—构建层—应用层”的主干思路，但做了明显收缩：存储层保留 SQLite + Neo4j，构建层只保留文本抽取与 schema 生成，应用层只保留最小问答页，不做推荐、网络分析、复杂图谱可视化。

### 存储层

- SQLite：保存项目、配置、批次、源文本、原始响应、三元组、schema、导入日志、问答日志
- Neo4j：保存项目图谱

### 构建层

- txt 上传与切分
- 大模型抽取
- 结果解析与人工修正
- 实体类型/关系类型提取
- 图谱导入

### 应用层

- 项目与模型配置页
- 批量抽取页
- schema/导入页
- 问答页

职责：

- 导入图谱
- 重建图谱
- 查看导入结果

### 4）问答页

职责：

- 输入问题

------

## 3.2 后端模块

### 1）ConfigService

职责：

- 管理模型配置
- 屏蔽前端对 api_key 的直接暴露
- 提供抽取模型与问答模型配置读取

### 2）ProjectService

职责：

- 项目 CRUD
- 状态流转
- 项目删除时联动清理 SQLite 和 Neo4j

### 3）BatchService

职责：

- 解析 txt
- 按行生成批次与源文本记录
- 维护批次统计与进度

### 4）ExtractionWorker

职责：

- 异步逐条调用抽取模型
- 通过 LLM Client / Provider Adapter 选择合适的 provider 调用策略
- 保存原始响应
- 做有限容错清洗
- 解析 `output` 字段
- 生成三元组

### 5）TripleService

职责：

- 三元组 CRUD
- 手工修正后重解析
- 手工新增/删除/修改三元组
- 维护三元组导入状态

### 6）SchemaService

职责：

- 从当前有效三元组提取实体类型与关系类型
- 维护实体类型、关系类型的增删改
- 做删除引用检查
- 在类型/关系改名时同步更新三元组

### 7）GraphService

职责：

- 初始化 Neo4j 约束
- 项目图谱导入
- 项目图谱重建
- 项目图谱删除
- 导入前冲突检测

### 8）QAService

职责：

- 生成 Cypher
- 校验 Cypher
- 执行 Cypher
- 再次调用问答模型组织答案
- 记录问答日志

### 9）CypherValidator

职责：

- 只读校验
- 危险关键字拦截
- 检查是否包含 `project_id` 范围限制

### 10）LLMClient / ProviderAdapter

职责：

- 对外提供统一的模型调用接口
- 在内部按 provider / 模型选择 SDK、HTTP、流式聚合等实现策略
- 在抽取场景下负责消息拼装、兼容参数注入和结果标准化
- 统一生成详细错误诊断，至少包含：错误码、异常类型、请求 URL、模型名、状态码、响应片段

## 4. SQLite 数据库设计

## 4.1 通用约定

- 主键统一使用 `TEXT`，值为 UUID
- 时间字段统一使用 UTC ISO8601 字符串
- 布尔值使用 `INTEGER`，0/1 表示
- JSON 原文、请求体、响应体统一存为 `TEXT`
- SQLite 开启 `PRAGMA foreign_keys = ON`

------

## 4.2 表结构清单

### 4.2.1 `model_configs`

用于保存抽取模型与问答模型配置。

```text
model_configs
- id                TEXT PK
- config_type       TEXT NOT NULL   -- extract / qa
- name              TEXT NOT NULL
- base_url          TEXT NOT NULL
- api_key           TEXT NOT NULL
- model_name        TEXT NOT NULL
- timeout_seconds   INTEGER NOT NULL DEFAULT 60
- is_enabled        INTEGER NOT NULL DEFAULT 1
- created_at        TEXT NOT NULL
- updated_at        TEXT NOT NULL

Constraints:
- UNIQUE(config_type, name)
```

说明：

- api_key 明文存库，但前端回显时只显示掩码
- 后端日志禁止打印 api_key

------

### 4.2.2 `projects`

用于保存项目主信息与状态。

```text
projects
- id                TEXT PK
- name              TEXT NOT NULL UNIQUE
- description       TEXT
- extract_config_id TEXT NOT NULL FK -> model_configs.id
- qa_config_id      TEXT NOT NULL FK -> model_configs.id
- status            TEXT NOT NULL   -- draft / ready / initialized / imported
- created_at        TEXT NOT NULL
- updated_at        TEXT NOT NULL
- last_import_at    TEXT
```

说明：

- 项目创建后默认 `draft`
- 抽取配置和问答配置都绑定后置为 `ready`
- 图谱初始化成功后置为 `initialized`
- 图谱最新导入成功后置为 `imported`
- 若导入后再修改三元组或 schema，则回退到 `initialized`

------

### 4.2.3 `import_batches`

用于保存一次 txt 上传形成的批次。

```text
import_batches
- id                   TEXT PK
- project_id           TEXT NOT NULL FK -> projects.id
- file_name            TEXT NOT NULL
- instruction          TEXT
- total_lines          INTEGER NOT NULL
- valid_lines          INTEGER NOT NULL
- status               TEXT NOT NULL   -- uploaded / extracting / partial_success / success / failed
- success_count        INTEGER NOT NULL DEFAULT 0
- request_failed_count INTEGER NOT NULL DEFAULT 0
- parse_failed_count   INTEGER NOT NULL DEFAULT 0
- started_at           TEXT
- finished_at          TEXT
- created_at           TEXT NOT NULL
- updated_at           TEXT NOT NULL

Indexes:
- INDEX(project_id, status)
```

说明：

- `instruction` 在首次发起抽取时写入
- 后续重试失败项沿用同一 instruction
- 若需要不同 instruction，重新上传新批次

------

### 4.2.4 `source_records`

用于保存批次中每一行源文本的处理记录。

```text
source_records
- id                 TEXT PK
- project_id         TEXT NOT NULL FK -> projects.id
- batch_id           TEXT NOT NULL FK -> import_batches.id
- line_no            INTEGER NOT NULL
- input_text         TEXT NOT NULL
- request_payload    TEXT
- raw_response       TEXT
- cleaned_output_text TEXT
- request_status     TEXT NOT NULL   -- pending / running / success / failed
- parse_status       TEXT NOT NULL   -- pending / success / failed
- is_manual_edited   INTEGER NOT NULL DEFAULT 0
- retry_count        INTEGER NOT NULL DEFAULT 0
- error_message      TEXT          -- 结构化错误诊断文本，可包含 error_code / exception_type / request_url / model_name / status_code / response_excerpt
- created_at         TEXT NOT NULL
- updated_at         TEXT NOT NULL

Constraints:
- UNIQUE(batch_id, line_no)

Indexes:
- INDEX(batch_id)
- INDEX(batch_id, request_status, parse_status)
```

说明：

- `line_no` 保存原 txt 的原始行号
- 空行忽略，但有效行仍保留原始文件位置
- `cleaned_output_text` 是对 `output` 做“去代码块、去首尾空白”后的结果

------

### 4.2.5 `extracted_triples`

用于保存三元组明细。

```text
extracted_triples
- id                 TEXT PK
- project_id         TEXT NOT NULL FK -> projects.id
- batch_id           TEXT NOT NULL FK -> import_batches.id
- source_id          TEXT NOT NULL FK -> source_records.id
- triple_key         TEXT NOT NULL    -- hash(project_id|subject|predicate|object)
- subject            TEXT NOT NULL
- subject_type       TEXT NOT NULL
- predicate          TEXT NOT NULL
- object             TEXT NOT NULL
- object_type        TEXT NOT NULL
- status             TEXT NOT NULL    -- valid / deleted
- is_manual          INTEGER NOT NULL DEFAULT 0
- imported           INTEGER NOT NULL DEFAULT 0
- created_at         TEXT NOT NULL
- updated_at         TEXT NOT NULL

Indexes:
- INDEX(source_id)
- INDEX(project_id, triple_key)
- INDEX(project_id, subject, predicate, object)
```

说明：

- 不对三元组表做唯一约束，因为要保留多来源重复三元组
- `triple_key` 仅用于导入时去重与冲突检测
- 编辑过的或新增的三元组，`imported` 置为 0

------

### 4.2.6 `entity_types`

用于保存项目级实体类型集合。

entity_types

说明：

- 支持手工新增

### 4.2.7 `relation_types`

用于保存项目级关系类型集合。

relation_types

### 4.2.8 `graph_import_logs`

用于保存图谱导入和重建日志。

## 5. Neo4j 图模型设计

## 5.1 设计原则

一期采用**统一标签 + 属性表达 schema**，不使用动态 Label 和动态关系类型。

原因：

- 跨领域适配更简单
- Cypher 生成规则更统一
- schema 变化时不需要频繁改图库结构
- 代码量更少

------

## 5.2 节点模型

### 节点标签

```text
:KGEntity
```

### 节点属性

```text
- project_id    STRING
- name          STRING
- entity_type   STRING
- created_at    STRING
- updated_at    STRING
```

### 节点唯一键

```text
(project_id, entity_type, name)
```

说明：

- 同名不同类型允许共存
- 不同项目中的同名实体完全隔离

------

## 5.3 关系模型

### 关系类型

```text
:RELATED_TO
```

### 关系属性

```text
- project_id       STRING
- predicate        STRING
- first_batch_id   STRING
- first_source_id  STRING
- first_line_no    INTEGER
- created_at       STRING
- updated_at       STRING
```

说明：

- 关系语义由 `predicate` 属性表达
- 只保存“首次导入来源”作为最小追踪字段
- 完整来源追溯保存在 SQLite 中

------

## 5.4 Neo4j 约束

建议在服务启动时执行一次：

```cypher
CREATE CONSTRAINT kg_entity_unique IF NOT EXISTS
FOR (n:KGEntity)
REQUIRE (n.project_id, n.entity_type, n.name) IS UNIQUE;
```

可选索引：

```cypher
CREATE INDEX kg_entity_project_name IF NOT EXISTS
FOR (n:KGEntity) ON (n.project_id, n.name);

CREATE INDEX kg_entity_project_type IF NOT EXISTS
FOR (n:KGEntity) ON (n.project_id, n.entity_type);
```

------

## 5.5 图谱导入规则

### 导入前去重

导入服务先从 SQLite 中取当前项目的所有 `status=valid` 三元组，再按：

```text
(project_id, subject, predicate, object)
```

做业务去重。

### 类型冲突检测

若同一 `triple_key` 对应多个不同的 `(subject_type, object_type)`，判定为**类型冲突**，该三元组不导入，并写入 `graph_import_logs.error_message` 或 `api_error_logs`。

### 节点写入

```cypher
MERGE (s:KGEntity {
  project_id: $project_id,
  entity_type: $subject_type,
  name: $subject
})
ON CREATE SET s.created_at = $now
SET s.updated_at = $now

MERGE (o:KGEntity {
  project_id: $project_id,
  entity_type: $object_type,
  name: $object
})
ON CREATE SET o.created_at = $now
SET o.updated_at = $now
```

### 关系写入

```cypher
MERGE (s)-[r:RELATED_TO {
  project_id: $project_id,
  predicate: $predicate
}]->(o)
ON CREATE SET
  r.first_batch_id = $batch_id,
  r.first_source_id = $source_id,
  r.first_line_no = $line_no,
  r.created_at = $now
SET r.updated_at = $now
```

说明：

- 关系 dedupe 实际由“端点 + predicate”保证
- 在导入前已保证同一 `(subject,predicate,object)` 不存在类型冲突

------

## 5.6 重建与删除

### 项目重建

先删除当前项目在 Neo4j 中的所有图数据，再重新导入：

```cypher
MATCH (n:KGEntity {project_id: $project_id})
DETACH DELETE n
```

### 项目删除

删除顺序建议如下：

1. 先删 Neo4j 当前项目图数据
2. 再删 SQLite 中该项目的所有元数据
3. 两步之间不做分布式事务
4. 如果第二步失败，保留错误日志并允许再次清理

## 7. 外部模型适配规则

## 7.1 认证方式

默认采用：

```http
Authorization: Bearer {api_key}
Content-Type: application/json
```

说明：认证头是统一约定，但具体调用入口、SDK 类型、URL 路径和流式能力由 provider 适配器决定。

## 7.2 LLM Client 内部统一接口

系统内部仍以统一抽象调用模型：

```json
{
  "model": "your-model-name",
  "instruction": "抽取或问答阶段的指令",
  "input": "单条文本或问句",
  "provider_options": {
    "temperature": 0.6,
    "top_p": 0.95,
    "max_tokens": 256
  }
}
```

说明：该结构是**系统内部接口**，不是要求所有真实 provider 直接接受的 HTTP 请求体。适配层可以把它转换为 SDK 调用、手写 HTTP 调用、流式聚合调用或其他 provider 特定格式。

## 7.3 抽取场景提示词拼装规则

抽取场景默认采用以下规则：

1. 将 `instruction + input_text` 合并为一条 `user` 消息。
2. `system` 消息不是强制方案，只能作为 provider / 模型兼容策略的一部分。
3. 若某 provider / 模型存在已验证参数，应允许按 provider / 模型注入，例如：`temperature`、`top_p`、`thinking_budget`、`max_tokens`。
4. 真实 provider 调用允许使用官方 / 兼容 SDK，不要求必须手写 HTTP。
5. 真实调用允许使用流式响应后聚合，不要求必须 `stream=False`。

## 7.4 抽取结果标准化

无论底层调用采用 SDK、HTTP 还是流式聚合，进入解析层前都应标准化为可落库文本。对于结构化抽取，系统以如下形式进入后续解析：

```json
{
  "output": "[{"subject":"...","subject_type":"...","predicate":"...","object":"...","object_type":"..."}]"
}
```

## 7.5 问答模型请求

问答场景仍可复用同一 LLM Client 抽象：

```json
{
  "model": "your-model-name",
  "instruction": "问答阶段后端拼装的指令",
  "input": "问句或问句+查询结果"
}
```

问答场景同样允许由适配层决定是 SDK、HTTP 还是流式聚合实现。

## 7.6 已验证兼容策略与推荐模型（截至 2026-03-18）

- 真实 provider 推荐验收组合：`SiliconFlow + deepseek-ai/DeepSeek-V3.2`
- `deepseek-ai/DeepSeek-V3.2` 适合作为当前结构化三元组抽取的推荐模型
- `deepseek-ai/DeepSeek-R1` 在新兼容方案下可返回成功结果，但输出更偏解释型，不推荐作为“稳定输出 JSON 数组”的结构化抽取模型
- SiliconFlow + DeepSeek 系列的抽取任务允许配置兼容参数，例如：
  - `temperature=0.6`
  - `top_p=0.95`
  - `extra_body.thinking_budget=128`
  - 对 JSON / 三元组 / Cypher 类结构化任务设置 `max_tokens=256`

## 8. 关键流程时序

## 8.1 txt 上传与批量抽取

1. 前端上传 txt 到 `/projects/{project_id}/batches/upload`
2. 后端按每行切分，忽略空行，创建 `import_batches` 和 `source_records`
3. 前端调用 `/batches/{batch_id}/extract`，传入 instruction
4. 后端将批次置为 `extracting`
5. 后端后台任务逐条处理：
   - 读取一条 `source_record`
   - 生成内部统一抽取语义 `{model, instruction, input}`
   - 默认将 `instruction + input_text` 合并为单条 `user` 消息
   - 由 `llm_client.py` 按 provider / 模型选择 SDK、HTTP、流式聚合和兼容参数
   - 保存标准化后的 `raw_response`
   - 从响应中取 `output`
   - 进行有限清洗
   - 尝试解析 JSON 数组
   - 生成 `extracted_triples`
   - 若失败，保存详细错误诊断
6. 前端轮询 `/batches/{batch_id}/progress`
7. 全部结束后，批次进入 `success / partial_success / failed`

------

## 8.2 原始响应修正与三元组编辑

1. 用户打开某条 `source_record`
2. 前端展示：
   - 输入文本
   - 原始响应
   - 已解析三元组
3. 若 `parse_status=failed`：
   - 用户修改 `raw_response`
   - 调用 `/sources/{source_id}/reparse`
4. 用户可新增、编辑、删除三元组
5. 任一三元组修改后：
   - 对应 `extracted_triples.imported = 0`
   - 若项目当前为 `imported`，回退到 `initialized`

------

## 8.3 Schema 刷新与维护

1. 用户点击“刷新 schema”
2. 后端扫描当前项目所有 `status=valid` 三元组
3. 汇总 `subject_type / object_type / predicate`
4. 将缺失项补充进 `entity_types / relation_types`
5. 不自动删除旧项
6. 用户可进一步增删改类型和关系

------

## 8.4 图谱初始化与导入

1. 用户点击“初始化图谱”
2. 后端检查 Neo4j 可连接
3. 后端确保全局约束存在
4. 项目状态置为 `initialized`

导入时：

1. 用户点击“导入图谱”
2. 后端读取当前项目所有有效三元组
3. 按 `(subject,predicate,object)` 做业务去重
4. 检查是否有同 key 不同 type 的冲突
5. 对无冲突三元组逐条 `MERGE`
6. 记录导入日志
7. 将项目状态置为 `imported`
8. 将相关三元组 `imported = 1`

------

## 8.5 图谱问答

1. 用户输入问题
2. 后端读取当前项目 schema
3. 组装“Cypher 生成指令”
4. 调用问答模型生成 Cypher
5. `CypherValidator` 执行安全校验
6. 校验通过后执行 Neo4j 查询
7. 返回原始命中结果
8. 后端组装“答案生成指令”
9. 再次调用问答模型生成自然语言答案
10. 写入 `qa_logs`
11. 返回 `generated_cypher + query_result + answer`

------

## 10. 状态机设计

## 10.1 项目状态

```text
draft
  -> ready         （绑定抽取配置和问答配置）
ready
  -> initialized   （初始化图谱成功）
initialized
  -> imported      （图谱导入成功）
imported
  -> initialized   （三元组或 schema 被修改，图谱变脏）
```

状态解释：

- `draft`：项目刚创建
- `ready`：可上传和抽取
- `initialized`：图谱已初始化，或需要重新导入
- `imported`：当前图谱和当前项目数据一致

```text
uploaded
  -> extracting

  -> success
extracting
  -> partial_success
extracting
  -> failed

```

## 11. 异常处理规则

## 11.1 txt 上传异常

- 非 txt 文件：返回 `400 INVALID_FILE_TYPE`
- 文件为空：返回 `400 EMPTY_FILE`
- 解析后无有效行：返回 `400 NO_VALID_LINES`

------

## 11.2 抽取模型调用异常

- 超时：`MODEL_TIMEOUT`
- 401/403：`MODEL_AUTH_FAILED`
- 5xx：`MODEL_SERVER_ERROR`
- 连接 / 网络异常：`MODEL_NETWORK_ERROR`
- provider / 模型 / 消息结构 / 传输策略兼容问题：`MODEL_PROVIDER_COMPAT_ERROR`

处理策略：

- 当前 `source_record` 记为 `request_failed`
- 不中断整个批次
- 允许单条重试和失败项重试
- 不做自动重试
- `error_message` 或错误日志中至少保留：错误码、异常类型、请求 URL、模型名、状态码、响应片段
- 真实 provider 验收必须使用“实际抽取提示词 + 实际目标输出格式”，不能只做简单问答 smoke test

------

## 11.3 输出解析异常

- 响应中无 `output`：`OUTPUT_FIELD_MISSING`
- `output` 非字符串：`OUTPUT_TYPE_INVALID`
- 清洗后仍不是合法 JSON 数组：`INVALID_JSON_OUTPUT`

处理策略：

- 保存 `raw_response`
- `parse_status=failed`
- 前端允许手工修正并重解析

------

## 11.4 三元组编辑异常

- 任一字段为空：`TRIPLE_FIELD_REQUIRED`
- 删除不存在的三元组：`TRIPLE_NOT_FOUND`

处理策略：

- 阻止保存
- 返回字段级错误提示

------

## 11.5 Schema 异常

- 删除仍被引用的实体类型：`ENTITY_TYPE_IN_USE`
- 删除仍被引用的关系类型：`RELATION_TYPE_IN_USE`
- 改名后名称重复：`SCHEMA_NAME_DUPLICATED`

处理策略：

- 返回 `409 CONFLICT`

------

## 11.6 图谱导入异常

- 项目未初始化：`GRAPH_NOT_INITIALIZED`
- Neo4j 不可连接：`GRAPH_UNAVAILABLE`
- 同一 triple_key 存在类型冲突：`TRIPLE_TYPE_CONFLICT`
- Cypher 执行异常：`GRAPH_IMPORT_FAILED`

处理策略：

- 导入日志记失败
- 不回滚 SQLite 中的三元组编辑结果
- 项目状态不置为 `imported`

------

## 11.7 问答异常

- 问答模型调用失败：`QA_MODEL_FAILED`
- 生成 Cypher 非法：`CYPHER_NOT_READONLY`
- 生成 Cypher 未包含项目范围：`PROJECT_SCOPE_MISSING`
- 查询为空：`NO_RESULT`

处理策略：

- `cypher_invalid`：不执行 Neo4j
- `no_result`：返回固定答案“当前知识图谱中未检索到相关信息。”
- `failed`：返回错误信息并记录日志

------

## 12. 最小实现建议

## 12.1 后端执行方式

本地单机场景建议：

- `uvicorn --workers 1`
- 批量抽取用 FastAPI 后台任务或单线程 worker
- 不引入 Celery / Redis
- 真实 provider 适配优先使用官方 / 兼容 SDK（例如 `openai==2.28.0`）
- 允许在客户端做流式响应聚合

## 12.2 导入策略

一期只支持：

- 项目级全量增量导入
- 项目级重建导入

不支持：

- 图谱内自动删除单条关系
- 图谱与 SQLite 的实时双向同步

## 12.3 前端交互策略

为了减代码量：

- 抽取进度使用**轮询**
- 不用 WebSocket
- Schema 不自动实时刷新，用户手工点击“刷新 schema”
- Cypher 只展示，不支持手工编辑执行
