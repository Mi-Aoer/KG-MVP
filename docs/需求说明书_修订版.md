# 通用领域知识图谱构建与问答系统（MVP）需求说明书

版本：V1.1
更新日期：2026-03-18
定位：一期最小可用版本
目标：以最低限度的功能和代码量，实现“**txt 批量导入 → 大模型关系抽取 → 抽取结果修正 → schema 生成 → Neo4j 导入 → 基于图谱问答**”的完整闭环。

------

## 一、文档目的

本文档用于明确系统一期的业务边界、功能需求、数据要求、页面要求和验收标准，作为后续开发、测试和迭代的参考依据。

本系统面向“可适配多个领域的知识图谱快速构建与问答”场景设计，强调以下原则：

1. **最小化实现**：只做完成业务闭环所必需的功能。
2. **领域无关**：系统不预置固定领域实体类型和关系类型，所有类型均从抽取结果中动态提取。
3. **可替换模型**：抽取模型 API 和问答模型 API 分离配置。
4. **可人工干预**：当模型输出不规范时，允许前端修正后继续流程。
5. **严格基于图谱问答**：问答结果必须以图数据库检索结果为依据，不允许脱离图谱自由发挥。
6. **兼容性优先**：真实 provider 只要求“可兼容接入”，不假设所有 OpenAI-compatible provider、模型、消息结构和传输方式都完全等价。

------

## 二、建设范围

### 2.1 本期范围

本期系统需要完成以下能力：

1. 上传 txt 文件并按“每行一条”切分为多条待处理文本。
2. 在前端填写本次抽取使用的 instruction。
3. 调用可配置的抽取模型 API，并允许后端按 provider / 模型兼容策略组装真实请求。
4. 接收模型返回结果，并展示原始响应与解析后的三元组结果。
5. 从模型返回的三元组中提取实体类型和关系类型。
6. 支持用户对三元组、实体类型、关系类型进行人工编辑。
7. 根据当前项目的实体类型和关系类型形成项目级 schema。
8. 在同一个 Neo4j 库中，按 `project` 维度组织和隔离图谱数据。
9. 一键将三元组导入图数据库，并按规则去重。
10. 提供单轮问答页面，通过通用大模型生成 Cypher，查询 Neo4j，并基于命中结果组织答案。
11. 展示问答过程中生成的 Cypher 和命中的原始查询结果。

### 2.2 本期不做

为保证系统简洁，一期不包含以下功能：

1. 多用户、登录、权限管理、审批流。
2. 高级图谱可视化页面。
3. 推荐系统、网络挖掘、统计分析大盘。
4. 多轮对话记忆。
5. 多租户、复杂组织级隔离。
6. 自动同步、版本回滚、复杂数据治理流程。
7. 系统内置大量模板化领域知识库。

------

## 三、系统角色

一期默认按**单用户/管理员**场景设计，不区分复杂角色。
系统使用者需要能够完成以下操作：

1. 创建项目。
2. 配置抽取模型 API 和问答模型 API。
3. 上传 txt 文件并发起抽取。
4. 修正模型返回结果。
5. 管理 schema。
6. 导入 Neo4j。
7. 发起图谱问答。

------

## 四、核心术语定义

### 4.1 项目（Project）

系统中的业务隔离单位。一个项目对应一套独立的抽取结果、schema 和图谱问答范围。

### 4.2 批次（Batch）

一次 txt 上传形成的导入批次。一个批次包含多条源文本记录。

### 4.3 源文本（Source Text）

txt 文件中的一行内容，对应一次抽取输入的 `input`。

### 4.4 三元组（Triple）

模型输出的关系记录，字段固定为：

- `subject`
- `subject_type`
- `predicate`
- `object`
- `object_type`

### 4.5 实体类型（Entity Type）

由三元组中的 `subject_type` 和 `object_type` 聚合得到的类型集合。

### 4.6 关系类型（Relation Type）

由三元组中的 `predicate` 聚合得到的关系集合。

### 4.7 项目级 Schema

当前项目使用的实体类型集合与关系类型集合，是该项目图谱结构的逻辑定义。

------

## 五、总体业务流程

系统整体业务流程如下：

1. 用户创建项目。
2. 用户配置抽取模型 API 和问答模型 API。
3. 用户上传 txt 文件，系统按每行一条切分，空行忽略。
4. 用户输入本次抽取的 instruction。
5. 系统对每条文本调用抽取模型接口；后端保存 `instruction` 与 `input`，并可按 provider / 模型兼容策略将其转换为真实请求。
6. 系统保存原始响应，解析 `output` 字段中的字符串化 JSON 数组。
7. 系统展示原始响应和解析后的三元组。
8. 若响应格式不合法，用户可手工修正原始响应或直接手工编辑三元组。
9. 系统从所有有效三元组中提取实体类型和关系类型，生成当前项目的 schema 初稿。
10. 用户可对实体类型和关系类型进行增加、删除、改名；也可继续调整三元组。
11. 用户确认 schema 后，系统在 Neo4j 中完成该项目图谱的初始化。
12. 用户一键导入三元组到图谱，系统按规则去重并记录来源信息。
13. 用户进入问答页面，输入问题。
14. 系统调用问答模型生成 Cypher。
15. 系统执行 Cypher 查询 Neo4j。
16. 系统展示生成的 Cypher 和命中的原始结果。
17. 系统基于查询结果生成最终答案；若无命中，则返回“未检索到相关信息”。

## 六、功能需求

## FR-01 项目与配置管理

### 需求说明

系统必须支持项目创建和基础配置管理。每个项目至少包含：

- 项目名称
- 项目说明（可选）
- 抽取模型配置绑定
- 问答模型配置绑定

系统必须支持两套独立模型配置：

1. **抽取模型配置**
2. **问答模型配置**

每套配置至少包含以下字段：

- `base_url`
- `api_key`
- `model_name`
- `timeout`
- `provider_options`（可选，用于保存 provider / 模型兼容参数）

### 业务规则

1. 抽取模型配置与问答模型配置必须独立保存。
2. 配置保存后，前端不应明文回显完整 `api_key`。
3. 模型调用必须由后端发起，不能由浏览器直接请求第三方模型接口。
4. 每个项目必须显式绑定一套抽取配置和一套问答配置后，才能进入完整流程。
5. 真实 provider 的调用层必须允许按 provider / 模型做兼容适配，不能要求所有 OpenAI-compatible provider 共用一套固定消息结构和固定传输策略。
6. `provider_options` 为可选扩展字段，用于保存温度、top_p、thinking_budget、max_tokens 等 provider / 模型级兼容参数。

### 验收标准

1. 可新增、编辑项目。
2. 可为项目选择抽取配置和问答配置。
3. 配置保存后可成功用于后续抽取与问答。

------

## FR-02 txt 批量导入

### 需求说明

系统必须支持上传 txt 文件作为抽取输入源。txt 文件处理规则如下：

1. 按“每行一条”切分。
2. 空行自动忽略。
3. 保留原始行号，用于后续定位来源。
4. 每次上传形成一个独立批次。

### 业务规则

1. 一条源文本对应一次抽取输入。
2. 每条源文本必须保存以下最小信息：
   - 所属项目
   - 所属批次
   - 行号
   - 原始文本
   - 当前处理状态
3. 系统一期仅要求支持 txt 文件，不要求上传 JSON 文件。

### 验收标准

1. 上传 txt 后可正确按行切分。
2. 页面能展示批次总行数和有效行数。
3. 能看到每条源文本的行号和原文。

------

## FR-03 抽取模型调用

### 需求说明

系统必须支持对当前批次中的每条文本调用抽取模型接口。

系统内部仍以“本次抽取 instruction + 单条源文本 input”为抽取语义，但真实 provider 的调用方式必须允许由后端兼容层按 provider / 模型进行适配，不要求所有 provider 都直接接受同一份固定 HTTP 请求体。

请求规则如下：

- `instruction` 由前端本次填写，并随批次保存
- `input` 为当前源文本内容
- 默认提示词拼装规则为：将 `instruction + input_text` 合并为**单条 user 消息**发送
- `system` 消息不是强制方案，只能作为可选的 provider 策略
- 真实 provider 调用允许使用官方 / 兼容 SDK，也允许使用手写 HTTP，但以稳定性优先
- 真实调用允许使用流式响应并在客户端聚合最终文本，不要求必须 `stream=False`
- 若某 provider / 模型存在已验证兼容参数，系统必须允许按配置或内置策略注入

### 建议内部调用参数

```json
{
  "model": "your-model-name",
  "instruction": "这里是用户填写的抽取指令",
  "input": "这里是一条源文本",
  "provider_options": {
    "temperature": 0.6,
    "top_p": 0.95,
    "max_tokens": 256
  }
}
```

说明：上面的结构是**系统内部抽取语义**，真实 provider 请求可由兼容层转换为 SDK 调用、流式聚合调用或其他 provider 适配格式。

### 期望响应格式

系统在落库和后续解析时，必须能够拿到统一的抽取结果文本。对于结构化三元组抽取，最终需满足：

- 返回结果中包含 `output` 字段；
- `output` 为字符串；
- 字符串内容是一个 JSON 数组。

例如：

```json
{
  "output": "[{"subject":"...","subject_type":"...","predicate":"...","object":"...","object_type":"..."}]"
}
```

### 业务规则

1. 抽取过程按“每条源文本一请求”执行。
2. 一期以**稳定优先**为原则，默认可采用串行处理。
3. 单条请求失败或解析失败，不得中断整个批次。
4. 系统必须保存每条文本的：
   - 请求内容
   - 原始响应
   - 调用状态
   - 错误信息（如有）
5. 错误信息不能只保留错误码；至少应保留：错误码、异常类型、请求 URL、模型名、状态码、响应片段（如有）。
6. `instruction` 必须随批次保存，便于追溯。
7. 真实 provider 验收不能只验证“最简单测试请求返回 200”，必须验证“实际抽取提示词 + 实际目标输出格式”。
8. 对真实 provider，设计上必须允许按 provider / 模型做兼容适配，不能继续假设所有 OpenAI-compatible provider 完全等价。

### 验收标准

1. 可对一个批次发起抽取任务。
2. 每条文本均可看到请求状态与详细错误信息。
3. 批次中部分记录失败时，其余记录仍能正常完成。
4. mock 链路与至少一组已验证的真实 provider / 模型组合都能跑通真实抽取。

### 当前推荐模型（截至 2026-03-18）

- 结构化三元组抽取推荐：`deepseek-ai/DeepSeek-V3.2`
- `deepseek-ai/DeepSeek-R1` 不推荐作为“稳定输出 JSON 数组”的结构化抽取模型

## FR-04 抽取结果展示与人工修正

### 需求说明

系统必须展示每条源文本对应的模型原始响应和解析后的三元组结果。

当模型返回的 `output` 不是合法 JSON 数组时，系统必须：

1. 展示原始响应；
2. 标记该条记录为“解析失败”；
3. 允许用户在前端手工修正后继续。

### 三元组编辑要求

用户必须可以逐条对三元组执行以下操作：

1. 新增
2. 编辑
3. 删除

每条三元组至少包含以下字段：

- `subject`
- `subject_type`
- `predicate`
- `object`
- `object_type`

### 业务规则

1. 当 `output` 为 `[]` 时，视为合法结果，表示当前文本未抽取出三元组。
2. 当原始响应无法解析时，用户可修正原始响应后重新解析。
3. 用户也可跳过原始响应解析，直接手工新增三元组。
4. 三元组任一关键字段为空时，不允许导入图谱。

### 验收标准

1. 页面可同时展示原始响应和结构化三元组。
2. 对不合法 JSON 可进行人工修正并重新解析。
3. 对任意三元组可进行增删改。

------

## FR-05 实体类型与关系类型提取及维护

### 需求说明

系统必须从有效三元组中自动提取项目级 schema 初稿：

- 实体类型集合 = 所有 `subject_type` 与 `object_type` 的去重并集
- 关系类型集合 = 所有 `predicate` 的去重集合

系统必须支持用户对实体类型和关系类型进行：

1. 增加
2. 删除
3. 改名

### 业务规则

1. 类型和关系名称不得写死在系统中。
2. 用户对类型或关系进行“改名”时，系统必须同步更新所有引用该名称的三元组。
3. 为降低误操作风险，若某实体类型或关系类型仍被三元组引用，系统默认**不允许直接删除**；用户需先修改或删除引用该项的三元组，再删除该类型/关系。
4. 用户手工新增的类型或关系，必须可在三元组编辑器中被选用。

### 验收标准

1. 系统能自动提取并展示实体类型与关系类型列表。
2. 用户改名后，相关三元组字段同步变化。
3. 删除被引用的类型或关系时，系统会阻止并提示原因。

------

## FR-06 项目级 Schema 生成与 Neo4j 初始化

### 需求说明

系统必须根据当前项目确认后的实体类型和关系类型，形成一份项目级 schema，并在 Neo4j 中完成该项目图谱的初始化。

由于一期采用“同一个 Neo4j 库按项目隔离”的方案，因此“创建图数据库”在本期定义为：

1. 在同一个 Neo4j 数据库中为当前项目建立可识别、可隔离的数据组织方式；
2. 创建项目元数据；
3. 初始化该项目所需的最小 schema 信息；
4. 为后续导入和问答提供当前项目范围约束。

### 业务规则

1. 系统级 Neo4j 连接配置由部署侧提供，不要求每个项目重复配置。
2. 每条节点和关系必须能够明确归属到某个 `project_id`。
3. 问答查询必须限制在当前项目范围内。
4. 一期允许采用“逻辑 schema”实现，即：
   - 在系统侧维护实体类型集合与关系类型集合；
   - 在 Neo4j 中按统一结构存储节点和关系，同时使用 `project_id`、`entity_type`、`predicate` 等信息实现项目隔离和类型约束。

### 验收标准

1. 可为项目执行“初始化图谱”操作。
2. 初始化成功后，项目进入“可导入”状态。
3. 不同项目数据在查询时不会混淆。

------

## FR-07 三元组导入图谱

### 需求说明

系统必须支持将当前项目中的有效三元组一键导入 Neo4j。

### 导入前校验

导入前系统必须检查：

1. 当前项目已完成 schema 初始化；
2. 所有待导入三元组字段完整；
3. 当前项目内不存在明显的结构冲突；
4. 所属项目和批次信息完整。

### 去重规则

用户确认的一期去重规则为：

- 对同一项目内相同的 `(subject, predicate, object)` 做精确匹配去重。

### 来源保留要求

系统必须保留以下来源信息：

- 来源批次
- 来源文本
- 来源行号

为减少图数据库复杂度，一期允许将完整来源追溯信息保存在业务库中，而图数据库中仅保留必要的来源标识。

### 业务规则

1. 导入成功后，系统应返回导入结果摘要，例如：
   - 创建节点数
   - 创建关系数
   - 去重忽略数
   - 失败数
2. 同一批次重复导入时，不应产生重复三元组。
3. 一期不要求实现“导入后自动双向同步删除”。若用户在导入后大幅修改三元组，建议通过“重建项目图谱”或“重新导入项目”方式处理。

### 验收标准

1. 可一键导入当前项目的有效三元组。
2. 相同 `(subject, predicate, object)` 不重复导入。
3. 可追溯导入关系对应的批次和原始文本来源。

------

## FR-08 图谱问答

### 需求说明

系统必须提供单轮问答页面，问答流程固定为：

1. 用户输入问题；
2. 系统基于当前项目 schema 调用问答模型生成 Cypher；
3. 系统执行 Cypher 查询 Neo4j；
4. 系统展示生成的 Cypher 和命中的原始结果；
5. 系统再次调用问答模型，根据查询结果组织自然语言答案。

### 业务规则

1. 问答必须基于当前项目，不允许跨项目查询。
2. 系统必须向问答模型提供当前项目的 schema 信息。
3. 生成的 Cypher 必须是只读查询。
4. 系统必须对 Cypher 做安全校验，禁止写操作和危险语句。
5. 问答模型生成答案时，必须严格基于命中的图谱结果。
6. 当图谱中无命中结果时，系统返回固定提示，例如：
   - “当前知识图谱中未检索到相关信息。”
7. 一期仅支持单轮问答，不保存上下文记忆。

### 展示要求

问答页面至少展示：

- 用户问题
- 生成的 Cypher
- 查询命中结果
- 最终答案

### 验收标准

1. 用户输入问题后，可生成并展示 Cypher。
2. 可执行图谱查询并返回原始命中数据。
3. 最终答案与命中结果保持一致。
4. 无结果时返回固定提示，不编造答案。

------

## 七、数据要求

## 7.1 业务元数据存储

除 Neo4j 外，系统需要一套轻量元数据存储，用于保存项目、批次、原始响应、三元组、schema 和问答日志。
一期可采用 SQLite 或 MySQL，优先推荐 SQLite 以降低复杂度。

建议至少包含以下数据对象：

1. **Project**
   - `project_id`
   - `project_name`
   - `description`
   - `status`
   - `extract_config_id`
   - `qa_config_id`
   - `created_at`
2. **ModelConfig**
   - `config_id`
   - `config_type`（extract / qa）
   - `base_url`
   - `api_key`
   - `model_name`
   - `timeout`
   - `provider_options`（可选，保存 provider / 模型兼容参数）
   - `enabled`
   - `created_at`
3. **ImportBatch**
   - `batch_id`
   - `project_id`
   - `file_name`
   - `instruction`
   - `total_lines`
   - `valid_lines`
   - `status`

## 7.2 Neo4j 图存储要求

一期图存储要求以“简单、可查、可隔离”为原则。

### 必须满足的能力

1. 能区分项目归属。
2. 能表示实体与关系。
3. 能按当前项目范围做问答查询。
4. 能从图中还原实体类型和关系类型信息。

### 一期建议实现

为减少动态 schema 带来的复杂度，一期建议采用以下轻量方式：

- 节点统一使用基础标签，例如 `KGEntity`
- 节点属性至少包含：
  - `project_id`
  - `name`
  - `entity_type`
- 关系统一使用基础关系，例如 `RELATED_TO`
- 关系属性至少包含：
  - `project_id`
  - `predicate`
  - `source_batch_id`
  - `source_line_no`

这是一种**逻辑 schema**实现方案。
项目级 schema 由系统维护，问答时由模型根据 schema 信息生成相应查询条件。

这样做的优点是：

1. 代码量低；
2. 适配不同领域更简单；
3. 不需要频繁生成大量动态标签和关系类型；
4. 更利于后续扩展。

## 九、非功能需求

### 9.1 简洁性

1. 系统不做非必要页面和流程。
2. 页面优先采用单项目多步骤或多标签页方式实现。
3. 后端逻辑优先保证可读性和可维护性。

### 9.2 可扩展性

1. 抽取模型和问答模型应通过配置替换。
2. 领域实体类型和关系类型不得写死。
3. 后续应能在当前基础上扩展更多导入源、问答能力和图算法功能。
4. 真实 provider 调用层必须允许按 provider / 模型做兼容适配，不得将“OpenAI-compatible”简单等同为“请求格式完全一致”。

### 9.3 安全性

1. 第三方模型 API Key 不得在浏览器侧直接暴露。
2. 所有外部模型调用均通过后端代理发起。
3. 生成的 Cypher 必须进行只读校验。
4. 禁止执行写库、删除、调用危险过程等操作。

### 9.4 可追溯性

1. 每条源文本需保留原始输入、原始响应、解析结果和处理状态。
2. 每个批次需保留 instruction。
3. 每次问答需记录问题、Cypher、命中结果和答案。
4. 真实 provider 调用失败时，错误诊断至少应保留：错误码、异常类型、请求 URL、模型名、状态码、响应片段。

### 9.5 性能

一期不追求高并发和大规模性能优化，以“可稳定跑通完整流程”为优先目标。
默认允许采用串行抽取、流式响应后聚合以及同步导入方式。

### 9.6 模型与 provider 兼容性约束

1. “OpenAI-compatible”只代表接口大体兼容，不代表所有 provider、所有模型、所有消息结构、所有传输方式都可完全等价替换。
2. 推理模型不一定适合结构化抽取任务；当要求严格 JSON 输出时，应优先选择已验证稳定的结构化输出模型。
3. 真实 provider 验收不能只测“简单问答能返回 200”，必须测“实际抽取提示词 + 实际解析目标格式”。

------

## 十、异常处理要求

### 10.1 抽取接口异常

1. 请求失败时记录错误信息，不中断整个批次。
2. 错误信息不能只保留简单错误码，必须保留至少：错误码、异常类型、请求 URL、模型名、状态码、响应片段（如有）。
3. 用户可看到失败记录，并能够据此区分“认证 / 网络 / 超时”与“provider 兼容性”问题。

### 10.2 输出格式异常

1. `output` 缺失或不是合法 JSON 字符串时，标记解析失败。
2. 系统必须展示原始响应并允许人工修正。

### 10.3 导入异常

1. schema 未初始化时，禁止导入。
2. 存在字段缺失的三元组时，禁止导入并提示具体问题。

### 10.4 问答异常

1. 生成的 Cypher 非法时，不得执行。
2. 查询结果为空时，返回固定提示。
3. 问答模型生成内容与查询结果不符时，以查询结果为准。

## 十一、验收标准

一期验收以完整业务闭环跑通为准，至少满足以下场景：

### 场景 1：批量抽取（功能验收 + 真实 provider 验收）

#### 功能验收

1. 创建项目并绑定抽取模型与问答模型配置。
2. 上传一个 txt 文件。
3. 系统按每行切分生成批次。
4. 输入 instruction 并发起抽取。
5. 系统能逐条返回原始响应、三元组结果、request_payload 和详细错误信息。
6. 批次状态、失败重试和部分成功场景正常。

#### 真实 provider 兼容性验收

1. 至少在一组已验证模型上跑通真实抽取。
2. 当前推荐验收组合：`SiliconFlow + deepseek-ai/DeepSeek-V3.2`。
3. 不再要求 `deepseek-ai/DeepSeek-R1` 作为结构化抽取验收模型。

### 场景 2：异常修正

1. 某些文本返回非法 JSON。
2. 用户可以在页面中看到原始响应。
3. 用户修正后可重新解析，或手工补录三元组。

### 场景 3：Schema 管理

1. 系统能从三元组中自动提取实体类型和关系类型。
2. 用户可新增、删除、改名。
3. 改名后相关三元组同步更新。

### 场景 4：图谱导入

1. 用户完成 schema 初始化。
2. 点击导入后，系统将三元组写入 Neo4j。
3. 相同 `(subject, predicate, object)` 不重复导入。
4. 可查看导入结果摘要。

### 场景 5：图谱问答

1. 用户进入问答页面并输入问题。
2. 系统生成 Cypher 并展示。
3. 系统执行查询并展示命中结果。
4. 系统基于结果返回最终答案。
5. 若无数据命中，则返回固定提示。

## 十二、一期建议实现边界

以下内容不是强制需求，但适合作为最小实现方案：

1. **后端**：FastAPI 或 Flask，优先选一个即可，不做复杂微服务拆分。
2. **前端**：简单单页应用或服务端模板页面均可，优先减少代码量。
3. **元数据存储**：SQLite。
4. **图数据库**：Neo4j 单实例。
5. **抽取流程**：串行调用，确保稳定；真实 provider 允许采用 SDK 流式聚合。
6. **问答流程**：单模型完成“Cypher 生成 + 答案组织”两步。
7. **图谱展示**：系统内部不做复杂图形展示，调试阶段可借助 Neo4j Browser。
8. **抽取推荐模型**：截至 2026-03-18，推荐使用 `deepseek-ai/DeepSeek-V3.2`。
9. **不推荐模型**：`deepseek-ai/DeepSeek-R1` 不建议作为严格 JSON 结构化抽取的默认推荐模型。
10. **真实 provider 验收方式**：必须使用实际抽取提示词和实际目标格式做联调验证。

------

## 十三、结论

本期系统的目标不是做一个功能完备的平台，而是做一个**可跑通、可修改、可扩展**的最小知识图谱构建与问答闭环。
其核心链路应严格聚焦在以下 6 步：

**txt 批量导入 → 大模型抽取 → 结果修正 → schema 确认 → Neo4j 导入 → 图谱问答**

只要这一链路稳定可用，一期目标即视为达成。

下一步最适合继续补的是两份开发材料：**数据库表结构清单**和**后端接口清单**。
