# 三份文档逐条修订对照表

生成日期：2026-03-18

## 1.《需求说明书》修订对照

| 序号 | 位置 | 原文/原意 | 修订后 | 修订原因 |
|---|---|---|---|---|
| 1 | 总体原则、建设范围、总体流程 | 默认把真实模型调用理解为“按 `instruction + input` 直接请求统一接口”。 | 明确新增“兼容性优先”原则：真实 provider 只要求可兼容接入，不假设所有 OpenAI-compatible provider、模型、消息结构、传输方式完全等价。 | 修正“OpenAI-compatible = 完全等价”的错误前提。 |
| 2 | FR-01 项目与配置管理 | 模型配置只有 `base_url / api_key / model_name / timeout`。 | 增加可选字段 `provider_options`，用于保存 provider / 模型兼容参数。 | 让 SiliconFlow 等场景下的温度、top_p、thinking_budget、max_tokens 等兼容参数有正式落点。 |
| 3 | FR-01 业务规则 | 未要求 provider 级兼容适配。 | 明确真实 provider 调用层必须允许按 provider / 模型做兼容适配，不能要求所有 OpenAI-compatible provider 共用一套固定消息结构和固定传输策略。 | 把这次问题的真正根因写进需求层，避免后续再按“统一固定格式”实现。 |
| 4 | FR-03 抽取模型调用 | 请求规则写成“每次请求固定包含 `instruction` 和 `input`”，且默认对应同一 HTTP 请求体。 | 改成“系统内部仍以 instruction + input 为抽取语义，但真实 provider 请求由后端兼容层按 provider / 模型适配，不要求所有 provider 直接接受同一份固定 HTTP 请求体”。 | 区分“系统内部抽取语义”和“真实 provider 传输格式”。 |
| 5 | FR-03 提示词拼装 | 未约束消息拼装策略，默认容易被实现成固定 system + user 结构。 | 明确默认规则为：将 `instruction + input_text` 合并成单条 `user` 消息；`system` 消息不是强制方案，只能作为可选 provider 策略。 | 解决 SiliconFlow/DeepSeek 场景下 system 消息不稳的问题。 |
| 6 | FR-03 调用方式 | 容易推导成必须手写 HTTP、必须非流式。 | 明确真实 provider 允许使用官方/兼容 SDK，也允许流式响应后在客户端聚合，不要求必须 `stream=False`。 | 将本次已验证有效的 SDK + 流式聚合方案写入需求。 |
| 7 | FR-03 兼容参数 | 原文没有 provider 级参数策略。 | 增加“若某 provider / 模型存在已验证兼容参数，系统必须允许按配置或内置策略注入”。 | 为 SiliconFlow + DeepSeek 系列兼容参数提供需求依据。 |
| 8 | FR-03 错误信息 | 错误信息只要求保存，粒度偏粗。 | 明确错误信息不能只保留错误码，至少应保留：错误码、异常类型、请求 URL、模型名、状态码、响应片段。 | 避免兼容性问题被误判成纯网络错误。 |
| 9 | FR-03 验收标准 | 仅要求能发起抽取、能看到请求状态、部分失败不影响整体。 | 增加“每条文本均可看到请求状态与详细错误信息”“mock 链路与至少一组已验证真实 provider/模型组合都能跑通”。 | 把“真实 provider 兼容性验收”纳入需求验收。 |
| 10 | FR-03 推荐模型 | 原文未区分推荐/不推荐模型。 | 明确截至 2026-03-18，结构化三元组抽取推荐 `deepseek-ai/DeepSeek-V3.2`，`DeepSeek-R1` 不推荐用于稳定 JSON 数组抽取。 | 把独立验证结论正式写入需求。 |
| 11 | 业务元数据定义 | `model_config` 无 provider 兼容参数；`error_message` 只是普通错误文本。 | 在数据要求中补充 `provider_options`；将 `error_message` 明确为可容纳详细诊断信息。 | 确保数据设计与实际排障需要一致。 |
| 12 | 风险说明 / 约束 / 场景验收 | 原文未充分强调 provider 兼容性风险。 | 新增或强化：OpenAI-compatible 仅代表大体兼容；推理模型不一定适合严格 JSON 抽取；真实 provider 验收必须用“实际抽取提示词 + 实际目标格式”；功能验收与真实 provider 验收分开。 | 将这次踩坑经验沉淀为产品级风险说明。 |

## 2.《系统设计文档》修订对照

| 序号 | 位置 | 原文/原意 | 修订后 | 修订原因 |
|---|---|---|---|---|
| 1 | 1.3 实现原则 | 只强调“所有外部模型调用都经由后端代理”。 | 增加三条设计原则：LLM Client 采用“统一接口 + provider / 模型适配器”；抽取默认单条 `user` 消息；真实 provider 允许官方/兼容 SDK 与 `stream=True` 聚合。 | 从系统层修正旧实现假设。 |
| 2 | 3.2 后端模块 - ExtractionWorker | 仅写“异步逐条调用抽取模型、保存原始响应、解析 output”。 | 增加“通过 LLM Client / Provider Adapter 选择合适的 provider 调用策略”。 | 明确兼容适配职责不在 worker 里硬编码，而在 client 层。 |
| 3 | 3.2 后端模块新增 | 原文没有专门的 LLM 兼容层模块。 | 新增 `LLMClient / ProviderAdapter` 模块，职责包括：统一接口、按 provider/模型选 SDK/HTTP/流式策略、消息拼装、兼容参数注入、结果标准化、详细错误诊断。 | 把本次修复方案抽象成正式架构组件。 |
| 4 | 4.2.4 source_records.error_message | 原字段只是 `TEXT`。 | 明确 `error_message` 为结构化错误诊断文本，可包含 `error_code / exception_type / request_url / model_name / status_code / response_excerpt`。 | 落库字段语义与排障需求对齐。 |
| 5 | 7.1 外部模型接口约定 | 原文容易让人理解成固定认证头 + 固定 HTTP 入口即可。 | 增补说明：认证头可统一约定，但调用入口、SDK 类型、URL 路径、流式能力由 provider 适配器决定。 | 避免设计文档继续暗示“统一手写 HTTP”方案。 |
| 6 | 7.2 LLM Client 内部统一接口 | 原文没有系统内部接口与 provider 外部接口的区分。 | 新增内部统一接口，加入 `provider_options`，并强调该结构只是系统内部接口，不要求真实 provider 原样接受。 | 统一“内部语义接口”和“外部真实调用”的边界。 |
| 7 | 7.3 抽取场景提示词拼装规则 | 原文未定义消息拼装规则。 | 新增规则：`instruction + input_text` 合并为单条 `user` 消息；`system` 仅为可选 provider 策略；允许注入 `temperature / top_p / thinking_budget / max_tokens`；允许 SDK 与流式聚合。 | 将本次真实兼容方案设计化。 |
| 8 | 7.4 抽取结果标准化 | 原文没有特别强调“流式聚合后再标准化”。 | 明确无论 SDK / HTTP / 流式聚合，进入解析层前都应标准化为统一可落库文本。 | 保证解析层与底层调用解耦。 |
| 9 | 7.6 已验证兼容策略与推荐模型 | 原文没有“已验证组合”和“推荐模型”。 | 增加：真实 provider 推荐验收组合为 `SiliconFlow + deepseek-ai/DeepSeek-V3.2`；`DeepSeek-V3.2` 适合当前结构化抽取；`DeepSeek-R1` 不推荐做稳定 JSON 抽取；允许 SiliconFlow + DeepSeek 系列兼容参数。 | 把验证结论升级成设计约束。 |
| 10 | 抽取链路时序 / 处理流程 | 原文默认调用步骤偏统一固定格式。 | 改成由 `llm_client.py` 按 provider / 模型选择 SDK、HTTP、流式聚合和兼容参数，失败时保存详细错误诊断。 | 让流程图和实现说明与真实代码改动一致。 |
| 11 | 异常分类 | 原异常更容易把问题归到 `MODEL_NETWORK_ERROR`。 | 增加 `MODEL_PROVIDER_COMPAT_ERROR`，明确区分连接/网络异常与 provider/模型/消息结构/传输策略兼容问题。 | 让异常模型能反映真实根因。 |
| 12 | 异常处理要求 | 原文未强制记录详细错误上下文。 | 明确 `error_message` 或错误日志至少保留：错误码、异常类型、请求 URL、模型名、状态码、响应片段；真实 provider 验收必须用真实抽取提示词和真实目标格式。 | 提升系统可观测性和验收有效性。 |
| 13 | 最小实现建议 / 结论 | 原文未列出兼容层。 | 将“LLM Client / Provider Adapter 兼容层”列为核心开发对象，并明确真实 provider 适配优先使用官方/兼容 SDK（例如 `openai==2.28.0`），允许客户端流式聚合。 | 让后续实现方向不再回退到旧方案。 |

| 1 | 开发前硬规则 第 5 条 | 写成“所有模型调用都走统一 client”，并隐含“OpenAI 兼容风格 client”。 | 改成“所有模型调用都走统一接口，但允许 provider / 模型适配”，不得假设所有 OpenAI-compatible provider 共享同一消息结构、SDK 行为和传输策略。 | 从任务规则层纠正旧实现方向。 |
| 2 | T01 依赖安装 | 未包含 `openai` SDK。 | 新增 `openai==2.28.0`。 | 与新方案保持一致。 |
| 3 | T06 目标 | 只要求“启动抽取任务 -> 后台逐条请求模型 -> 保存原始响应 -> 更新进度”。 | 改成“在此基础上，将真实 provider 接入从固定 HTTP 请求升级为统一接口 + provider / 模型兼容适配”。 | 把修复内容纳入任务目标，不再是临时补丁。 |
| 4 | T06 统一模型 client 接口 | 原实现方向容易落成所有 provider 共用一份 HTTP 请求体。 | 明确“统一的是服务内部调用接口，不是要求所有真实 provider 接受同一份 HTTP 请求体”。 | 从任务说明上避免开发者再次误读。 |

| 6 | T06 provider 调用方式 | 原文容易继续手写 `httpx.post` 非流式请求。 | 明确允许官方/兼容 SDK、允许 `stream=True` 后客户端聚合，不要求继续硬写非流式 HTTP。 | 把有效修复方案落实到任务层。 |
| 7 | T06 provider 兼容参数 | 原文没有 provider 级参数适配要求。 | 明确允许按 provider / 模型注入 `temperature / top_p / thinking_budget / max_tokens`。 | 支撑 SiliconFlow + DeepSeek 系列兼容调用。 |
| 8 | T06 推荐真实验收组合 | 原文无指定。 | 明确当前推荐验收组合：`SiliconFlow + deepseek-ai/DeepSeek-V3.2`，且 `DeepSeek-R1` 不再作为结构化抽取默认验收模型。 | 将真实联调结论沉淀为任务标准。 |
| 9 | T06 llm_client 代码骨架 | 原文更像 mock + 简单 HTTP 调用。 | 改成“Mock + ProviderAdapter + 统一错误诊断”结构，并加入 `build_provider_options(...)`、流式聚合、兼容错误分类等骨架。 | 让开发骨架直接朝新方案实现。 |
| 10 | T06 错误落库 | 原文失败分支更像 `source.error_message = str(e)`。 | 改成优先保存结构化错误 JSON；真实 provider 兼容问题不再只压成 `MODEL_NETWORK_ERROR`，而要带出 `MODEL_PROVIDER_COMPAT_ERROR` 与详细诊断。 | 解决排障信息不足问题。 |

| 12 | T06 验收清单 | 原文只看进度、raw_response、失败可重试。 | 增加真实 provider 下错误信息字段检查，以及“已验证组合能完成一次真实结构化抽取并得到可解析结果”。 | 提高验收的真实性。 |

| 15 | 最终交付 / 回归要求 | 原文只要求 mock 和真实 API 都能跑。 | 明确真实 provider 验收至少覆盖一组已验证组合（当前推荐：`SiliconFlow + deepseek-ai/DeepSeek-V3.2`）。 | 避免“真实 API 能返回 200”就被误判为已完成。 |

## 4. 一句话总结

这三份文档的修订方向是一致的：

- 从“统一固定 HTTP 调用”改成“统一内部语义 + provider / 模型兼容适配”
- 从“system 消息可默认依赖”改成“默认单条 user 消息，system 仅为可选策略”
- 从“非流式手写请求”改成“允许 SDK + 流式聚合”
- 从“只记简单错误码”改成“保留可定位的结构化错误诊断”
- 从“mock 跑通即可”改成“mock 功能验收 + 真实 provider 兼容性验收”
- 从“DeepSeek-R1 可默认验收”改成“结构化抽取默认推荐 DeepSeek-V3.2”
